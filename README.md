# StegShield

StegShield is a bachelor thesis project for detecting suspicious and dangerous image files using a hybrid approach:

- Binary CNN-based steganography evidence estimation.
- Metadata and file-structure inspection.
- A fusion layer that labels files as `safe`, `suspicious`, or `dangerous`.

The first version will be an installable command-line tool:

```bash
stegshield analyze ./image.png
```

Later versions will expose the same analysis pipeline through an API so applications can scan uploaded images, such as profile pictures or user-generated content.

Full technical documentation for the thesis (architecture decisions, experiments,
numbers, insights, references) lives in
[docs/THESIS_DOCUMENTATION.md](docs/THESIS_DOCUMENTATION.md).

## Current Status

The project has a metadata analyzer CLI, binary CNN training/evaluation, and an optional fusion path for combining CNN stego probability with metadata risk indicators.

## Planned Features

- File type validation.
- Hashing and file metadata extraction.
- EXIF and structural risk indicators.
- Binary custom CNN trained from scratch for `clean` / `stego` detection.
- Optional comparison against pretrained CNN models.
- Combined CNN + metadata decision layer for final `safe`, `suspicious`, and `dangerous` labels.
- CLI output in human-readable and JSON formats.
- Future FastAPI integration.

## Labels

- `safe`: no suspicious indicators.
- `suspicious`: weak or moderate risk indicators.
- `dangerous`: strong evidence of payload-like behavior, steganographic embedding, malformed structure, or multiple high-risk indicators.

## Dataset Preparation

Place images in the labeled folders:

```text
data/raw/safe/
data/raw/suspicious/
data/raw/dangerous/
```

Then create train/validation/test CSV files:

```bash
stegshield prepare-dataset
```

This writes:

```text
data/splits/train.csv
data/splits/val.csv
data/splits/test.csv
```

If the dataset already provides official train/validation/test folders, preserve those
splits instead of reshuffling them. The `prepare-dataset` command refuses to overwrite
existing split CSV files unless `--force` is passed.

For the Kaggle Stego Images Dataset, use the dedicated importer after organizing files
under `data/raw/{safe,suspicious,dangerous}/kaggle_stegoimages/`:

```powershell
stegshield import-kaggle-splits
```

This writes portable relative paths and preserves the dataset split structure:

```text
data/splits/train.csv
data/splits/val.csv
data/splits/test_standard.csv
data/splits/test_adversarial.csv
data/splits/test_full.csv
```

Use `test_standard.csv` for normal comparable evaluation, `test_adversarial.csv` for
extra encoded/archive stego variants, and `test_full.csv` only when you want both
groups together.

## Custom CNN Training

After installing ML dependencies and preparing the dataset:

```bash
stegshield train-cnn --epochs 5 --batch-size 16 --image-size 256 --device cuda
```

The image loader crops by default instead of downscaling, because resizing can destroy
pixel-level steganography artifacts. The crop is taken from the **top-left corner** by
default (`--crop top-left`): sequential LSB embedders, including the tool used by the
Kaggle Stego Images Dataset, write payload bits row by row starting at pixel (0, 0), so
a center crop can discard the entire stego signal (a short payload only modifies the
first image rows). Use `--crop center` for an ablation experiment.

Inputs are fed to the CNN as raw 0-255 pixel values by default
(`--normalization raw255`), matching the convention of published steganalysis CNNs
whose truncation thresholds (T=3, T=2) are defined on that scale.

The training command supports two CNN variants:

```powershell
stegshield train-cnn --task stego --model yedroudj
stegshield train-cnn --task stego --model steganalysis
```

The `yedroudj` model is Yedroudj-Net, a published spatial-steganalysis CNN used as the
literature baseline:

> M. Yedroudj, F. Comby, M. Chaumont, "Yedroudj-Net: An Efficient CNN for Spatial
> Steganalysis", IEEE ICASSP 2018, pp. 2092-2096, doi:10.1109/ICASSP.2018.8461438.

It uses a fixed 30-filter SRM high-pass preprocessing bank (unnormalized kernels), an
ABS activation in block 1, truncation activations (T=3, T=2) in blocks 1-2, average
pooling, global average pooling, and a 256/1024 fully connected classifier. Because
StegShield analyzes RGB images (the paper uses grayscale BOSSBase), the SRM bank is
applied per color channel via a grouped convolution.

The `steganalysis` model is StegShield's proposed variant: it shares the same fixed
30-filter SRM front-end (with TLU clipping) but replaces the plain convolutional
blocks with residual blocks, so the comparison isolates the feature extractor design.

Accuracy is secondary for this project because the binary split can be imbalanced
(`clean` may be much smaller than `stego`). Training selects the best checkpoint by
validation macro F1 by default, and reports balanced accuracy, majority-class baseline
accuracy, per-class recall, and a confusion matrix.

Class-weighted loss and balanced sampling are configurable:

```powershell
stegshield train-cnn --task stego --class-weights --balanced-sampler
stegshield train-cnn --task stego --no-class-weights --balanced-sampler
stegshield train-cnn --task stego --class-weights --no-balanced-sampler
```

Both are enabled by default for `train-cnn` because the current binary task is
imbalanced. For ablation experiments, run one technique at a time and compare macro F1,
balanced accuracy, clean recall, and stego recall.

### Training performance on small GPUs

If GPU memory usage is close to 100% (check `nvidia-smi`), the Windows driver
spills CUDA allocations into shared system RAM and training becomes 10-50x
slower while still showing "100% GPU utilization". On a 4 GB GPU (for example a
laptop RTX 3050), use a smaller batch plus mixed precision and DataLoader
workers:

```powershell
stegshield train-cnn --task stego --model steganalysis --epochs 15 `
  --batch-size 16 --amp --num-workers 4 --device cuda `
  --output-model outputs/models/steganalysis_stego.pt `
  --output-metrics outputs/reports/steganalysis_stego_training.json
```

Training now prints per-epoch progress with an ETA, saves the best checkpoint
immediately whenever validation improves, and rewrites the metrics JSON after
every epoch — so an interrupted run keeps its best model and its history.

If multiple NVIDIA GPUs are present, choose explicitly:

```powershell
stegshield train-cnn --device cuda:0
stegshield train-cnn --device cuda:1
```

The recommended CNN task is binary steganalysis:

```text
safe -> clean
suspicious/dangerous -> stego
```

The final `safe` / `suspicious` / `dangerous` label should be produced by fusing the
CNN stego probability with metadata and file-structure risk indicators.

The default outputs are:

```text
outputs/models/stegshield_cnn.pt
outputs/reports/training_metrics.json
```

Evaluate a saved checkpoint on a split:

```powershell
stegshield evaluate-cnn `
  --model-path outputs/models/stegshield_cnn.pt `
  --split-csv data/splits/test_standard.csv `
  --output-report outputs/reports/test_standard_metrics.json `
  --device cuda
```

The evaluation report includes accuracy, majority-class baseline accuracy, macro F1,
balanced accuracy, per-class precision/recall/F1, false negatives by class, and a
confusion matrix.

After training a binary CNN, `analyze` can produce a fused result:

```powershell
stegshield analyze image.png `
  --cnn-model-path outputs/models/steganalysis_cnn.pt `
  --device cuda
```

Evaluate the final hybrid system on a risk-labeled split:

```powershell
stegshield evaluate-fusion `
  --model-path outputs/models/steganalysis_cnn.pt `
  --split-csv data/splits/test_standard.csv `
  --output-report outputs/reports/fusion_test_standard_metrics.json `
  --device cuda
```

This report compares metadata-only, CNN-only binary stego evidence converted to risk,
and the fused final classifier.

## Payload-Size Regression Head (multi-task steganalysis)

The proposed `steganalysis` model can be extended with a second head that estimates the
sequential-LSB payload size (not just clean/stego). Build a synthetic ground-truth set
(payloads of known size embedded by StegShield's own embedder, so the CNN does not learn
from the statistical estimator), train multi-task, and feed the CNN estimate into fusion:

```powershell
stegshield make-payload-regression-set
stegshield train-cnn --task stego --model steganalysis --payload-head `
  --train-csv data/splits/regress_train.csv --val-csv data/splits/regress_val.csv `
  --output-model outputs/models/steganalysis_multitask.pt --amp --num-workers 4 --device cuda
stegshield evaluate-fusion --model-path outputs/models/steganalysis_multitask.pt `
  --split-csv data/splits/test_standard.csv --payload-source cnn `
  --output-report outputs/reports/fusion_payload_cnn.json --device cuda
```

`--payload-source` selects whether the LSB payload severity comes from the statistical
estimator (default), the CNN regression head (`cnn`), or `both`. The detection head is
unaffected, and detection-only checkpoints load unchanged. Full step-by-step experiments
(detection no-regression check, regression MAE, CNN-vs-statistical agreement, fusion
comparison) are in [docs/payload_head_experiments.md](docs/payload_head_experiments.md).

## Thesis Interpretation

The CNN is intentionally binary because the visual model should estimate whether image
pixels contain steganography-like evidence. It should not be presented as a malware
detector and it should not directly assign the final `safe`, `suspicious`, or
`dangerous` labels. Those final labels are produced by the fusion layer.

The baseline CNN is included as a control experiment. If the plain CNN collapses to one
class, that is still useful evidence: it shows that naive visual learning is not enough
for this problem. The steganalysis CNN is the proposed visual model because high-pass
residual filters and truncation emphasize small pixel residuals where hidden-data
artifacts are more likely to appear. Current early results suggest it improves over the
baseline, but CNN-only detection remains weak when stego recall is low.

Metadata and static file analysis are necessary because many suspicious image-container
properties are not visible in RGB pixels. Extension/type mismatches, embedded archive or
executable signatures, trailing bytes, parser errors, and suspicious metadata strings
are file-level indicators. The fusion layer combines this static evidence with CNN
stego probability to produce the final risk label.

The static analyzer includes a sequential-LSB payload estimator
(`stegshield/metadata/lsb_payload.py`): sequential LSB embedding leaves a leading
noise-like run in the LSB plane followed by an abrupt return to image-like statistics,
which both detects the embedding and estimates the payload size. The structural idea
follows classical sequential-LSB steganalysis:

> A. Westfeld, A. Pfitzmann, "Attacks on Steganographic Systems", Information Hiding
> 1999, LNCS 1768, pp. 61-76.

Payload size drives severity: script/binary-scale payloads (>= 128 bytes) are
high-severity, marker-scale payloads (addresses, URLs) are medium-severity. This gives
the fusion layer a real signal for separating `suspicious` from `dangerous`. Embedded
executable detection validates the PE header structure (e_lfanew pointer to
`PE\x00\x00`) instead of matching the two-byte "MZ" pattern, which appears by chance in
most compressed image data and previously caused ~21% of clean files to be escalated to
`dangerous`. The pre-fix fusion report is kept as an ablation in
`outputs/reports/fusion_test_standard_metrics_oldrules.json`.

Accuracy alone is not sufficient. With imbalanced clean/stego data, a model can reach
high accuracy by predicting only the majority class. Final reporting should emphasize
macro F1, balanced accuracy, per-class recall, confusion matrices, and false negatives.
For this security-oriented project, false negatives are especially important because
they represent suspicious or dangerous files missed by the system. False positives also
matter because excessive warnings reduce usability, but they are usually easier to
triage when the system provides explanations.

Final thesis claims should be modest:

- The steganalysis CNN can be compared against the baseline CNN as a stronger visual
  stego-evidence estimator.
- CNN-only detection is limited on the current dataset and should not be treated as the
  final detector.
- The main contribution is the hybrid evaluation pipeline comparing baseline CNN,
  steganalysis CNN, metadata-only analysis, CNN-only risk conversion, and fusion.

## Dataset Direction

Candidate datasets include:

- Kaggle Stego Images Dataset: https://www.kaggle.com/datasets/marcozuppelli/stegoimagesdataset
- StegoAppDB: https://forensicstats.org/stegoappdb/
- BOSSBase / BOWS2-based steganalysis datasets.
- ALASKA steganalysis datasets.

Real malicious files should not be executed. Suspicious samples must be handled defensively and, if necessary, in an isolated environment.
